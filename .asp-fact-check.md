## FC-001 — Claude Code `settings.local.json` 的 deny 合併行為

- **日期**：2026-06-08
- **觸發**：ADR-011（將 ASP 動態 deny 隔離至 settings.local.json）的核心假設
- **查證方式**：claude-code-guide subagent 對照官方文件（code.claude.com/docs/en/settings、/permissions）
- **結論（已確認）**：
  1. `.claude/settings.local.json` 是 Claude Code 自動載入的設定 scope。✅
  2. `permissions.deny` / `permissions.allow` **跨 scope 合併**；評估順序 **deny → ask → allow**，**任一 scope 的 deny 即阻擋**（deny-first，與檔案優先序無關）。✅
  3. 優先序：Managed > Command-line > **Local (settings.local.json)** > Project (settings.json) > User。✅
  4. `settings.local.json` **預設 gitignored**，定位為本地 / 不提交的 per-project override。✅
  5. Hook 寫入 settings.local.json 的 deny 同 session 自動 reload 生效，並持續至下次 session。✅
- **對 ADR-011 的意涵**：把 ASP 動態 deny 寫到 settings.local.json，其強制力與寫在 settings.json **等價**（deny-first、scope-agnostic），同時獲得 gitignored + 本地隔離 → 解決 TD-005 殘留。
- **再驗證條件**：Claude Code 若變更 settings 載入或 deny-merge 語意，FC-001 失效，須重審 ADR-011。

## FC-002 — Claude Code PreToolUse hook 介面（deny 機制 / input schema）

- **日期**：2026-06-12
- **觸發**：ADR-020 / SPEC-013（PreToolUse commit 閘）的核心實作假設
- **查證方式**：claude-code-guide subagent 對照官方文件（code.claude.com/docs/en/hooks）
- **結論（已確認）**：
  1. **settings.json 配置**：`hooks.PreToolUse[]`，每項含 `matcher`（工具名，regex/字串，`"Bash"` 匹配 Bash 工具）+ `hooks[]`（`type:"command"` + `command`）。✅
  2. **hook stdin JSON**：含 `tool_name`（如 `"Bash"`）、`tool_input.command`（Bash 指令字串）、`hook_event_name`、`cwd` 等。✅
  3. **Deny 機制（兩法）**：
     - 方式 A（採用）：`exit 0` + stdout JSON `{"hookSpecificOutput":{"hookEventName":"PreToolUse","permissionDecision":"deny","permissionDecisionReason":"..."}}` → 阻擋並把 reason 回饋 AI。`permissionDecision` 值域：allow/deny/ask/defer。
     - 方式 B：`exit 2` + stderr → 阻擋，stderr 回饋 AI。其他 exit code = 非阻擋。✅
- **不採用 / 不依賴**：agent 提到的 settings `if:"Bash(git commit *)"` 欄位**未在採用範圍**（不確定是否穩定）→ SPEC-013 改為 `matcher:"Bash"` + **hook 腳本內自行 grep command 判斷是否 git commit**（可控、不依賴未確認欄位）。
- **對 SPEC-013 的意涵**：commit 閘用方式 A（exit 0 + permissionDecision deny），腳本內解析 `tool_input.command`，非 git commit 一律 `defer`（不干擾其他 Bash）。
- **再驗證條件**：Claude Code 若變更 PreToolUse hook 的 stdin schema 或 deny 輸出格式，FC-002 失效，須重審 SPEC-013 hook 腳本。

## FC-003 — Claude Code 自訂 slash 指令的子目錄命名（`commands/asp/` → `/asp:xxx`）

- **日期**：2026-06-17
- **觸發**：安裝器同步 commands bugfix 採 namespaced 子目錄 `~/.claude/commands/asp/`；對抗式審查提出 blocker 主張子目錄不產生 `/asp:` 前綴（裝完叫 `/approve-adr`）。需定第三方行為。
- **查證方式**：
  1. WebFetch 官方文件 `code.claude.com/docs/en/slash-commands`（命名表 + namespacing 段，存檔於 tool-results）。
  2. 本 session live 實證：建立 `.claude/commands/asp/approve-adr.md` 後，harness 系統訊息實際將其註冊為 `asp:approve-adr`、`asp:review-work`（帶冒號）。
- **結論（已確認，針對本環境）**：
  1. `.claude/commands/<檔>.md`（直接位於 commands 下）→ 指令名 = 檔名去副檔名，例 `deploy.md` → `/deploy`。✅
  2. 冒號 namespace 文件主述為 plugin（`plugin-name:skill-name`）；但文件亦載「directory-qualified name」同樣用冒號（root `deploy` 與 `apps/web/.claude/skills/` 的 → 後者 `apps/web:deploy`）。
  3. **子目錄下的 command（`commands/asp/approve-adr.md`）在本環境實際註冊為 `asp:approve-adr`**（live 實證），故呼叫名為 `/asp:approve-adr`。審查員「冒號=plugin 限定」之推論被 live registration 推翻 → blocker 判為**誤報**。
- **殘留不確定**：使用者其他電腦若跑不同 Claude Code 版本，子目錄命名解析可能不同（最壞退化為 `/approve-adr`，無冒號）。緩解：**子目錄 `commands/asp/` 的檔案組織本身穩健**（安全整批同步的依據不變），僅「顯示/呼叫名是否帶冒號」會因版本而異。
- **再驗證條件**：Claude Code 若變更 `.claude/commands/` 子目錄的命名/namespacing 規則，FC-003 失效，須重審指令內文與 CHANGELOG 的呼叫名標示。

## FC-004 — Claude Code plugin / marketplace 機制（ADR-021 分發改造可行性）

- **日期**：2026-06-23
- **觸發**：ADR-021（採用官方 plugin marketplace 取代 29KB 自製 install.sh + 全域 sync）的可行性核心假設——ASP 強制力深綁 SessionStart + PreToolUse hook，須先證實官方 plugin 機制完整承載這些元件，否則改造會打掉護欄。
- **查證方式**：WebFetch 官方文件（`code.claude.com/docs/en/plugins`、`/plugins-reference`、`/plugin-marketplaces`，舊 `docs.anthropic.com` 已 301 遷移，全文存檔於 tool-results）。
- **結論（已確認）**：
  1. **Hook 事件**：`hooks/hooks.json`（plugin root）支援的事件含 **`SessionStart`**（plugins-reference L115「When a session begins or resumes」）與 **`PreToolUse`**（L119「Before a tool call executes. **Can block it**」）。→ ASP 的 L1（session-audit）與 L1.5（commit 閘）兩層強制力，plugin 機制原生承載。✅
  2. **腳本路徑**：`${CLAUDE_PLUGIN_ROOT}`（L635，plugin 安裝目錄絕對路徑）可在 hook command 引用 bundled 腳本。✅
  3. **元件打包**：plugin 可含 skills/、agents/、commands/、`hooks/hooks.json`、`.mcp.json`、`.lsp.json`、monitors/、bin/、settings.json；manifest = `.claude-plugin/plugin.json`（name/description/version/author）。✅
  4. **分發/安裝/更新**：`.claude-plugin/marketplace.json` 列 plugins → `/plugin marketplace add <owner/repo>`（GitHub 直接 host，含 private repo）；`version` 設則 bump 才更新，省略且走 git 則 **commit SHA = 版本**（每 commit 自動更新）；使用者 `/plugin marketplace update` 刷新。✅
  5. **官方/社群市集**：`claude-plugins-official`（Anthropic 策展）+ `claude-plugins-community`（`/plugin marketplace add anthropics/claude-plugins-community`，經審查）。✅
- **殘留不確定（兩點，列為 ADR-021 → Accepted 前的驗收條件）**：
  1. **hook 動態寫 `settings.local.json`**（ASP L2 動態 deny 的做法）官方文件未明載；但 plugin hook 以非沙箱 shell 執行（command 可任意 shell），技術上可寫檔。**需 POC 實證**。
  2. **plugin `settings.json` 目前只支援 `agent`/`subagentStatusLine` 兩鍵**（plugins 文件明載，未知鍵 silently ignored）→ ASP 動態 deny **不能**改走 plugin settings.json，必須留在 hook 端寫 settings.local.json（與殘留 1 同一驗收）。
  3. **skill 命名空間化**：plugin skill 一律 namespaced（`/plugin-name:skill`）→ ASP 現 `/asp <意圖>` router 改 plugin 後呼叫名會變（如 `/asp:...`），需評估對既有 docs/使用者肌肉記憶的影響（與報告 §2 mono-router 議題相關）。
- **整體判定**：核心承載（SessionStart + PreToolUse-deny + 打包 + 市集自動更新）**已證實，低風險高信心**；殘留集中在「動態 deny 的 hook 寫檔」與「命名空間遷移」兩點，皆可由 POC 收斂——故 ADR-021 為**低風險提案**而非探索性賭注。
- **再驗證條件**：Claude Code 若變更 plugin hook 事件集合、`${CLAUDE_PLUGIN_ROOT}`、marketplace.json schema 或 `/plugin marketplace add` 介面，FC-004 失效，須重審 ADR-021。

## FC-005 — addyosmani/agent-skills 結構與設計事實（深層借鏡報告依據）

- **日期**：2026-06-23
- **觸發**：深層借鏡研究報告（`docs/research/2026-06-23-addyosmani-agent-skills-deep-borrow.md`）對第三方 repo 的結構/設計宣稱須一手查證後方可寫入（外部事實驗證鐵則）。
- **查證方式**：WebFetch 一手原始檔——`raw.githubusercontent.com/addyosmani/agent-skills/main/README.md` 與 `/CONTRIBUTING.md`。
- **結論（已確認）**：
  1. **skill 數 = 24**：23 lifecycle + 1 meta `using-agent-skills`（後者「Maps incoming work to the right skill workflow and defines shared operating rules」）。✅
  2. **六階段 SDLC**：DEFINE / PLAN / BUILD / VERIFY / REVIEW / SHIP（+ Meta）。✅
  3. **設計原則（README 原文四條）**：「Process, not prose.」/「Anti-rationalization.」（每 skill 列藉口×反駁表）/「Verification is non-negotiable.」（「'Seems right' is never sufficient.」）/「Progressive disclosure.」（references「load only when needed, keeping token usage minimal.」）。✅
  4. **CONTRIBUTING 品質四標準**：Specific（"Actionable steps, not vague advice"）/ Verifiable（"Clear exit criteria with evidence requirements"）/ Battle-tested（"Based on real engineering workflows, not theoretical ideals"）/ Minimal（"Only the content needed to guide the agent correctly"）。✅
  5. **bundling 門檻**：「Don't create supporting files unless content exceeds 100 lines」——此 100 行為支援檔（scripts/references）分檔門檻，**非** PR 大小或單任務改檔上限（更正二手摘要誤植）。✅
  6. **frontmatter**：必填 `name` + `description`；description「starts with what the skill does (third person), then includes one or more `Use when` trigger conditions」。✅
  7. **marketplace 安裝**：`/plugin marketplace add addyosmani/agent-skills` + `/plugin install agent-skills@addy-agent-skills`。✅
  8. **slash 指令 8 個**：`/spec /plan /build /test /review /webperf /code-simplify /ship`。✅
  9. **授權**：MIT。✅
- **殘留不確定**：「每任務 ≤5 檔 = scope 過大警訊」此具體數字出自二手（部落格/個別 skill），README/CONTRIBUTING 一手未載 → 報告**不以一手事實斷言該數字**，僅借「把質性 scope 紀律量化為機械信號」之精神。star 數（≈65.7K）為動態統計非設計事實，僅作活躍度脈絡。
- **再驗證條件**：addyosmani/agent-skills 改版（skill 數/階段/原則/CONTRIBUTING 標準/marketplace 指令變動）時 FC-005 失效，須重審報告對應段落。

## TD-002 — bypass-log 截斷偵測邏輯語義錯誤 [IMPORTANT]

**位置：** `.asp/hooks/session-audit.sh` 第 56-67 行（Iron Rule B）  
**問題：** `LINE_COUNT`（bypass 事件行數）與 `GIT_LINE_COUNT`（git commit 次數）是不同單位，比較無法可靠偵測截斷

```bash
# 改為從 git 取得上次提交時的行數
PREV_LINE_COUNT=$(git show HEAD:"${NDJSON_LOG}" 2>/dev/null | wc -l || echo 0)
if [ "${LINE_COUNT}" -lt "${PREV_LINE_COUNT}" ]; then ...
```
**風險：** Iron Rule B 偵測效果低於預期，截斷攻擊可能未被偵測

## TD-004 — Draft 偵測 regex 過寬，可能誤報 [IMPORTANT]

**位置：** `.asp/hooks/session-audit.sh` 第 152 行  
**問題：** `` `Draft` `` backtick 模式會匹配 ADR 內文說明段落（如「`Draft`（初稿）→ `FIRM` → `Accepted`」），不只匹配表格狀態欄  
**修法：** 改為精確匹配表格欄位：
```bash
grep -qiE "\|\s*\`Draft\`\s*\|" "$adr_file"
```
**風險：** 若未來 Accepted ADR 內含流程說明樣板文字，可能被誤判為 Draft 並觸發 commit deny

## TD-005 — managed deny 可能誤刪使用者手動設定 [IMPORTANT]

**位置：** `.asp/hooks/session-audit.sh` Section 10 自清邏輯（第 284-303 行）  
**問題：** reconcile 邏輯依固定字串移除 MANAGED_DENY，若使用者手動加入相同的 `Bash(git commit *)` deny（出於其他原因），下次 session 無 Draft ADR 時會被靜默移除  
**修法：** 在 settings.json 中為 ASP 注入的 deny 加識別 comment，或改用獨立欄位 `asp_managed_deny`，reconcile 只清除自己注入的條目  
**風險：** 使用者手動設定的 deny 被靜默移除，使用者毫不知情

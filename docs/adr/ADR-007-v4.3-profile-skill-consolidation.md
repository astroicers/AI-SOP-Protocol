## 背景（Context）

v4.2.0 完成後，ASP 現有：
- **18 個 profile**：其中 `task_orchestrator`（1383 行）、`multi_agent`（436 行）、`pipeline`（506 行）三者互引嚴重，載入順序依賴隱性
- **20 個 skill**：其中 `asp-escalate`（159 行）與 `asp-handoff`（236 行）功能重疊（escalate 是 handoff 的特殊類型）；`asp-qa`（83 行）是 `asp-dev-qa-loop`（197 行）的子集；`asp-security`（71 行）邏輯已在 `asp-ship` Step 9 涵蓋

**問題一：Profile 三角依賴**

```
task_orchestrator.md
  └─ Part G: 呼叫 multi_agent_dispatch()（stub，實作在 multi_agent）
multi_agent.md
  └─ IF pipeline_active: 引用 pipeline G1-G6 門檻邏輯
pipeline.md
  └─ 依賴 task_orchestrator 的 execute_*() 函數
```

三者形成三角循環依賴，AI 載入任一個都需要理解另外兩個，維護時容易產生定義漂移。

合併 multi_agent 進 task_orchestrator 後，依賴圖**降維**為雙邊循環：
```
task_orchestrator（含原 multi_agent 邏輯）↔ pipeline
```
雙邊循環仍存在，但從三角降為雙邊，維護面積縮小。

**問題二：Skill 功能重疊**

| Skill | 問題 |
|-------|------|
| `asp-escalate` | 產生 ESCALATION YAML handoff，是 `asp-handoff` 類型之一 |
| `asp-qa` | 83 行薄層，僅是 `asp-dev-qa-loop` 的前半段 |
| `asp-security` | 71 行，`asp-ship` Step 9 已含 secret scan；OWASP 掃描邏輯無人引用 |

---

## 評估選項（Options Considered）

### 選項 A：全部合併（激進）

將 `task_orchestrator + multi_agent + pipeline` 合為單一 `orchestrator.md`；5 個 skill 合為 2 個。

**優點**：檔案數最少，依賴消除  
**缺點**：`orchestrator.md` 將超過 2000 行，AI context 載入成本反而更高；pipeline G1-G6 細節若埋在大檔中難以維護

### 選項 B：部分合併（本次選定）

- **Profile**：`multi_agent.md` 內容移入 `task_orchestrator.md` Part G（替換原有的引用），`pipeline.md` 保持獨立
- **Skill**：`asp-escalate` 合入 `asp-handoff`；`asp-qa` 合入 `asp-dev-qa-loop`；`asp-security` 刪除，邏輯移入 `asp-ship` Step 9

**優點**：消除最高重疊度的 `task_orchestrator ↔ multi_agent` 循環，`pipeline.md` 保留細節；skill 數從 20 → 18（-2 個）  
**缺點**：`task_orchestrator.md` 行數增加至估計 ~1739 行（詳見行數估算），接近 2000 行上限，未來應避免繼續擴增

### 選項 C：只刪不合（保守）

只刪 `asp-security`，其餘不動。

**優點**：風險最低  
**缺點**：三角依賴問題未解決，ADR-006 roadmap 沒有推進

---

## 決策（Decision）

選擇**選項 B**。

理由：
1. `task_orchestrator ↔ multi_agent` 重疊最嚴重，合併後 AI 載入路徑更清晰
2. `pipeline.md` 的 G1-G6 定義細節是獨立查閱需求，保持獨立符合單一職責
3. Skill 合併消除三個薄層（83 + 71 + 159 = 313 行），預期淨減 ~200 行

### 行數估算（逐段計算）

| multi_agent.md 區塊 | 行數 | 合入狀況 |
|--------------------|------|---------|
| Part G routing stub（task_orchestrator 已有） | ~80 行 | 不重複合入 |
| Worker 規則 / MCP 安全邊界 | ~120 行 | 合入 |
| Autonomous 模式整合 | ~60 行 | 合入 |
| Converge 邏輯 / DAG / 衝突偵測 | ~100 行 | 合入 |
| 角色分派 / 交接協議 | ~76 行 | 合入 |
| **合入小計** | **~356 行** | |

task_orchestrator.md 合入後估計行數：1383 + 356 = **~1739 行**

選項 A 門檻為 2000 行，選項 B 合入後 ~1739 行，差距 ~260 行。選項 B 在可接受範圍內（< 2000 行），但接近上限，未來 task_orchestrator 應避免繼續擴增。

### 影響評估

- **載入路徑**：`mode: multi-agent` 時，CLAUDE.md 目前載入順序為 `global_core + multi_agent + task_orchestrator + pipeline`；合併後改為 `global_core + task_orchestrator + pipeline`
- **破壞性**：無對外 API；現有 session 不受影響（每次 session 重新載入）
- **循環依賴**：三角循環降維為雙邊循環（task_orchestrator ↔ pipeline），無法完全消除，但維護面積縮小
- **估計淨減少**：~350 行（skill 刪除）- ~356 行（profile 合入）+ ~100 行（重複段落消除）= **~94 行淨減少**；主要效益是消除三角依賴，而非行數減少

---

## 替代方案未選原因

- **選項 A**：合併後單檔過大，AI 每次載入整個 orchestrator 的成本高於三檔分散
- **選項 C**：只刪 asp-security 無法解決三角依賴，ADR-006 v4.3 目標未達成

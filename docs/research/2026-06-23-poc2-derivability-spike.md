# POC-2 Spike：「可推導性」能否機械化？（ADR-022）

**探索性 spike，非生產代碼。** 回答 ADR-022 風險最高的一格：複雜度棘輪的「可推導性軸」（一條規則若能由其他規則推導＝冗餘＝砍除候選）能否以**便宜、確定性、exit-code 化的機械代理**近似——還是只能靠 LLM judge。

## 方法

探針 `poc2-derivability-probe.py`（與本檔同目錄）對 ASP 真實治理語料做「跨檔近似重複義務偵測」當低保真機械代理：
- 語料：`CLAUDE.md` + 12 個 profiles + 15 個 skills（28 檔）。
- 抽「義務句」（含 `必須/須/禁止/不可/一律/應/跳過/deny/must` 等標記）。
- 正規化（去 markdown、去 `ADR-NN`/`G1-6` 編號引用、去標點空白）後，char-trigram **Jaccard 相似度**找跨檔近似重複（門檻 0.55）。
- 假設：若「同義務多處重述」是治理過載主因，這個便宜代理該撈得出來。

## 結果

| 測量 | 值 |
|------|----|
| 抽出義務句 | 434 條 |
| 跨檔近似重複群（Jaccard≥0.55, 跨≥2 檔） | **3 組** |
| 其中真語意重複 | **1 組**（`git push origin main 鐵則禁止` × 2 檔） |
| 結構樣板假陽性 | **2 組**（`// 不可違反的約束` 註解頭 × 4 檔、`├── 依賴 global_core.md` 宣告行 × 2 檔）→ 精確度 ~33% |

**召回失敗（決定性）**：研究指認的主病「同義務三處重述」中最典型的「commit 前須有測試/ship」，在語料裡有至少三種**不同措辭**（`任何 git commit 前 → /asp-ship`、`commit 前跑測試 / asp-ship`、hook 機制描述），字面相似度低於門檻，探針**一條都沒撈到**。

## 結論

**可推導性無法用便宜的字面代理機械化。** 字面相似度同時有：
- **精確度問題**：撈到的多是結構樣板，非真冗餘（~33% 精確度）。
- **召回問題（更致命）**：真正的重述是「換句話說」，語意等價但字面不同 → 漏掉。

根因：**可推導性本質是語意關係**（X 的行為能否由 Y+Z 推導），字面比對抓不到。要真判斷只能靠 **LLM judge**——但即使包成 exit-code，它仍**非確定性、每 PR 成本高**，「機械判定」退化成「機械呼叫一個會飄的判官」，正是 ADR-020 點名「LLM 判定義務最先蒸發」的風險。

## 建議（觸發 ADR-022 內建退場條件）

- **POC-1 行數棘輪**：確定性、全自動、便宜 → **保留為機械 gate**。
- **可推導性/去重軸**：**從「每-commit 機械 gate」降級為「定期 advisory review」**（LLM-judge 輔助列去重候選給人類判斷，**不阻擋 commit**）。保住去重價值，不假裝它是硬 gate、不撞 ADR-020。

> 這正是 spike 的價值：花最小成本，在拍板前發現決策有一半不該做成機械 gate——比建完一個藏在 exit-code 裡的飄判官再回頭，省太多。對應 ADR-022 收窄為「行數棘輪為唯一機械 gate + 去重降 advisory」。

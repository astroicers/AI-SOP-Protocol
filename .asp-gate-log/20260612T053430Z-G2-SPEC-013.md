## 6. 對抗式技術可實作性檢驗（重點）

### Finding F1：ship-trace 新鮮度判定——只證明測試跑過，非完整 asp-ship 10 步 [WARN]

**問題核心**：

SPEC 用「`.asp-test-result.json` passed=true 且 mtime ≥ `.git/index`」作為 ship 痕跡。但：

- `.asp-test-result.json` 由 asp-ship **Step 10** 寫入（依 SPEC 文字）。
- asp-ship 10 步包含：Step 1 CHANGELOG、Step 2 ADR 狀態、Step 3 TODO、Step 4 Traceability、Step 5 CONTEXT、Step 6 commit message draft、Step 7 bypass log、Step 8 docs sync、Step 9 敏感資訊掃描、Step 10 痕跡寫入。
- **然而**，使用者也可能只跑 `make test`（僅跑測試，不走 10 步 skill）。SPEC 的 deny reason 說「請先跑 `/asp-ship`（或 `make test`）」——意即 `make test` 也被視為有效痕跡。

**實質風險**：若 `make test` 也能寫 `.asp-test-result.json passed=true`，hook 無法區分「完整 10 步 asp-ship」vs「只跑測試」。SPEC 以「**`make test` 後放行**」的 Done When 手動驗證條目，實際上**默許了這個近似**。

**SPEC 誠實程度**：SPEC 在 deny reason 文字及 Done When 都承認 `make test` 即可放行，但在目標段（Goal）說「把 commit 前 asp-ship 從散文義務升為硬強制」——這兩者存在語意落差（"asp-ship" ≠ "make test"）。SPEC **沒有誠實標注此近似的後果**（例如「Step 1-9 的非測試義務仍靠散文」）。

**建議**：在 Edge Cases 或 Out of Scope 明確說明：此 hook 只強制「測試通過」，Steps 1-9 的散文義務不在機械偵測範圍；與 ADR-020 的「誠實界定能力邊界」原則一致。

### Finding F2：`.git/index` mtime 可靠性——git add 行為 [WARN]

**問題**：`git add` 是否「每次」都更新 `.git/index` mtime？

**已知行為**：
- `git add <file>` 確實更新 `.git/index` 的 mtime（寫入新 index 後原子替換）。
- **但**：若工作目錄完全乾淨（無任何 unstaged 變更），`git add .` 可能為 no-op，`.git/index` mtime **不變**。
- **`git commit --amend`** 情境：amend 後 `.git/index` 的 mtime 視實作不定（amend 並不一定觸發 index 更新），`.asp-test-result.json` 若在 amend 前寫入，mtime 關係可能誤判為「新鮮」甚至「stale」。

**SPEC 處理**：Edge Cases 提到「`git commit --amend` → 視同 commit（同樣需 ship 痕跡）」，但**沒有說明 amend 時 `.git/index` mtime 行為是否可靠**。

**實質風險**：若 amend 不更新 `.git/index`，且 `.asp-test-result.json` 比最後一次 `git add` 新，hook 會放行 amend commit，即使自上次 ship 以來已有實質 staged 變更（透過 amend 追加）。

**建議**：補充說明 amend 情境下 `.git/index` mtime 的已知限制，或改用 `git diff --cached --stat` 的存在性而非 mtime 比較。

### Finding F3：fail-open 繞過——刪 jq 或破壞 hook [MEDIUM-RISK]

**問題**：jq 缺失 → `defer`（放行）。SPEC 說「hook 受 Iron Rule A 保護（納入 CRITICAL_FILE）」，但：

1. Iron Rule A（由 `session-audit.sh` 執行）**在 SessionStart 時**才檢查 hash。**在 session 中途**刪除 jq 或破壞 hook 腳本，當次 session 不會被 Iron Rule A 攔截（Iron Rule A 只在下次 session 啟動時報 BLOCKER）。
2. jq 是 OS 級別工具，不受 Iron Rule A 保護；AI 若在同一 session 跑 `apt-get remove jq` 或腳本層刪除，commit gate 當場失效為 fail-open。

**SPEC 的說明**：SPEC 說「fail-open：jq 不存在 / 腳本錯誤 → defer + stderr WARN，**絕不死鎖**（強制力讓位於可用性，誠實殘留：此時退回散文）」——這是**誠實承認**的設計取捨，但**未說明「在 session 內移除 jq 不受 Iron Rule A 保護」這個具體攻擊面**。

**結論**：fail-open 設計取捨本身合理（避免死鎖），但 SPEC 應明確說明 jq 不受 Iron Rule A 保護、session-mid 攻擊面為已知殘留風險。

### Finding F4：ASP_SHIP_OK escape hatch——bypass 留痕是否充分 [WARN]

**SPEC 說明**：`ASP_SHIP_OK=1` 放行 + 記 `SHIP-GATE bypass` 遙測至 `rule-hits.jsonl`。

**問題**：
1. `rule-hits.jsonl` 存放於 `~/.claude/asp/metrics/`（User-level，非 project repo）。根據既有 Memory 紀錄（Iron Rule B 的 log 是 gitignored），JSONL 型 bypass log **不進 git**，跨 session 截斷偵測依賴 HWM sidecar。此處的 `rule-hits.jsonl` bypass 記錄面臨相同問題：**遙測存在，但不在 git 歷史中可見**。
2. SPEC 說「誠實留痕，非無聲跳過」，但未說明 bypass 記錄是否有後續稽核機制（是否有 `make audit-bypasses` 類指令）。
3. 更重要的：**AI 能否直接在 commit 指令前環境注入 `ASP_SHIP_OK=1`**？若 AI 學到「設這個就過」，escape hatch 退化為標準繞過路徑。SPEC 沒有要求 bypass 需要人類介入或二次確認。

**建議**：明確說明 bypass 的審計可見性（rule-stats 可見即足夠，但需說明）；考慮說明「bypass 不應由 AI 自行決策使用，須使用者主動設定」。

### Finding F5：複合 command grep 誤判 [MEDIUM-RISK]

**SPEC 邊界條件**：「複合 command（`git add . && git commit -m ...`）→ grep 偵測到 `git commit` 即進入判定」。

**潛在誤判情境**：

1. **commit message 含 "git commit" 字串**：例如 `git commit -m "docs: remove git commit hook"` — grep `git commit` 會命中，進入判定邏輯。這是**合法 commit**，不應因 message 內容觸發誤判分支（雖然最終仍看 ship 痕跡決定 defer/deny，邏輯仍正確，但進入判定本身無害）。
2. **`git log --grep="git commit"`** 或 **`git log --all --oneline | grep "git commit"`**：此類非 commit 操作若以 Bash 呼叫，hook 會偵測到 `git commit` 字串，進入判定 → 但因 command 非 `git commit`（而是 `git log`），若 grep 只抓關鍵字而非解析指令，**可能誤判為 commit 操作**。

**實質影響**：若 hook 腳本用簡單 `grep 'git commit'` 而非「確認指令是 `git commit` 而非其他 git 子命令」，`git log --grep` 或 commit message 含關鍵字的情境可能觸發誤判。**SPEC 未說明 hook 會如何精確解析**（是 word-boundary grep 還是指令解析）。

**建議**：SPEC 應說明 hook 解析邏輯（如：抓取 command 的第一個 git subcommand，而非全文 grep），或在 Edge Cases 補充說明複合 command 的具體解析策略。

### Finding F7：ADR-020 為 Draft 狀態，SPEC-013 引用狀態相容性 [INFO]

ADR-020 目前狀態為 `Draft`（禁止實作）。SPEC-013 引用 ADR-020 作為關聯 ADR。

依 ASP 鐵則：「Draft ADR 禁止生產代碼；FIRM ADR 允許 commit」。

SPEC 本身是規格層（非實作），引用 Draft ADR 不違反此鐵則。**但 SPEC 實作（hook 腳本撰寫）必須等 ADR-020 升 FIRM 後才能 commit**。SPEC 文件本身未提醒此約束。

**建議**：在 SPEC Traceability 或 Out of Scope 補一行：「實作 commit 須等 ADR-020 升 FIRM」，避免實作者遺忘。

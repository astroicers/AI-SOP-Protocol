## 7. 技術正確性對抗式審查

### FIND-1（嚴重度：MEDIUM）：canonical hash 計算規範不足

**問題描述：**

SPEC 在「輸出規格」中定義 hash 計算為 `sha256(prev + canonical_without_h)`，在 Edge Cases 中補充「canonical 必須穩定（key 排序 + 緊湊）」，但整份 SPEC 從未給出：

1. 「key 排序」的定義：字典序（UTF-8 byte order）還是其他順序？
2. 「緊湊」的定義：`jq -c` 格式？空白如何處理？是否移除 `\n`？
3. `prev` 與 `canonical_without_h` 的串接方式：直接字串接合（無分隔符）？若 `prev` 本身包含 sha256 hex 字串（64 chars），直接接合是否足以避免 ambiguity？
4. 字串 vs bytes：sha256 的輸入是 UTF-8 encoded bytes？輸出是 lowercase hex？

**風險：** 若實作者與未來驗證者對「canonical」格式理解不一致（例如一個用 `jq -c -S`，另一個用 Python `json.dumps(sort_keys=True)`），**相同內容可能算出不同 hash**，導致假陽性 BLOCKER（chain 驗證無故失敗）。這在遷移腳本與 session-audit.sh 之間尤其危險：兩段代碼若 canonical 格式不同，既有 log 遷移後立刻 BLOCKER，完全無法使用。

**建議：** SPEC 必須明確規定 canonical 格式，至少給出 reference 實作的 shell 指令，例如：
```
canonical = $(echo "$entry" | jq -cS 'del(.h)') 
h = echo -n "${prev}${canonical}" | sha256sum | awk '{print $1}'
```

### FIND-2（嚴重度：HIGH）：容錯模式繞過漏洞（真實設計風險，SPEC 未處理）

**問題描述：**

SPEC 在輸出規格中定義：

> 舊格式（無 hash 欄，未遷移） → 不報 chain BLOCKER（容錯，提示遷移），HWM 仍生效

這個設計在 SPEC 中被合理化為「避免升級即假陽性」，但它產生了一個**真實的攻擊向量**：

**攻擊步驟：**
1. 攻擊者（或知道機制的 AI）對 `.asp-bypass-log.ndjson` 執行等量替換。
2. 同時移除全部 `prev`/`h` 欄位（將 log 還原為「舊格式」）。
3. session-audit 的 chain 驗證邏輯偵測到「無 hash 欄」→ 觸發容錯分支 → **不報 chain BLOCKER**，只提示遷移。
4. HWM 因行數不變，**也不報 BLOCKER**。
5. 等量替換無痕成功。

**這完全還原了 TD-002 要解決的威脅**，等同選項 D（維持現狀）。攻擊者只需在竄改後移除 hash 欄，就能把已有 chain 保護的 log 退化回 HWM-only 狀態，而 session-audit 不報任何 BLOCKER。

**SPEC 未處理此漏洞。** Edge Cases 章節確實提到舊格式容錯，但沒有評估「已遷移 log 被人為移除 hash 欄」這個情境。這是不同情境：
- 真正的舊格式：從未有 `h` 欄（升級前狀態）
- 偽舊格式：原有 `h` 欄後被移除（攻擊狀態）

這兩種情況從 log 文字無法區分——這是設計根本矛盾，因為 session-audit 無外部基準可知「這個 log 應該有 chain 還是不應有」。

**建議：**
SPEC 應明確說明此設計取捨，選擇以下一種策略：
- **策略 A（更強保護）**：一旦 log 中任意一筆有 `h` 欄，整份 log 進入「嚴格模式」，之後出現沒有 `h` 欄的 entry 視為異常 → BLOCKER。
- **策略 B（維持容錯，誠實承認殘留）**：在威脅模型章節明文記錄此設計的限制：「攻擊者可透過剝除 hash 欄將 chain 降級回 HWM-only，此為 local-only 無外部錨定下不可避免的殘留，與 ADR-019 誠實界定的 tamper-evidence 而非 tamper-proof 一致。」
- **策略 C（sidecar 記錄遷移時間戳）**：建立一個 gitignored sidecar（類似 HWM），記錄「本 log 已遷移至 chain 格式」的時間戳；session-audit 若在此時間戳之後看到無 hash 的 entry → BLOCKER。

無論選哪種，**SPEC 必須承認此風險並說明選擇理由**，否則實作者可能無意間把容錯邏輯寫得過於寬鬆。

### FIND-3（嚴重度：LOW-MEDIUM）：Rollback「向後相容」宣稱過度樂觀

**問題描述：**

SPEC 在 Rollback Plan 中宣稱：

> `prev`/`h` 為附加欄，舊 session-audit 忽略未知欄位 → 向後相容

這句話在技術上**可能正確，也可能不正確**，取決於：

1. session-audit.sh 目前解析 NDJSON 的方式。若代碼使用 `jq .h` 存取特定欄位、不存在時回傳 `null`，則確實向後相容。若使用 `awk` 或 `grep` 依欄位順序解析，多出的欄位可能破壞現有邏輯。
2. 「舊 session-audit 忽略未知欄位」是一個**假設**，SPEC 沒有引用任何代碼位置或測試案例來支撐此假設。
3. Rollback Plan 中「回滾已測試：否」自認未驗證，但然後宣稱「向後相容」——這是矛盾的。

**建議：** 將「向後相容」改為「預期向後相容（待 Rollback 測試實際驗證）」，或引用 session-audit.sh 的具體代碼行作為支撐。

---

### FIND-4（嚴重度：MEDIUM）：測試矩陣未完整涵蓋 ADR-019 成功指標

**問題描述：**

ADR-019 的成功指標共四項：
1. 等量替換偵測 100% → **N1 涵蓋** ✓
2. chain 斷裂偵測 100% → **N2 涵蓋** ✓
3. 正常 append 無誤報 0 false positive → **P1 涵蓋** ✓
4. 既有測試零回歸 → **P1/B2 + make test 指令涵蓋** ✓

表面上全部涵蓋，但仔細對照 Edge Cases：

- **遷移前不假陽性**（B2 描述「遷移前不假陽性」）→ 測試矩陣只有 B2「遷移後 chain 通過」，**遷移前的負向測試（舊格式不報 BLOCKER）**沒有獨立測試案例（P 或 B 欄位）。
- **HWM sidecar 竄改場景**（ADR-019 背景提到 HWM sidecar 自身被竄改作為 TD-002 殘留之一）→ 測試矩陣完全沒有 HWM 竄改場景。ADR-019 宣稱 hash chain 作為「與 HWM 獨立的第二驗證」，但測試矩陣未驗證「HWM 被竄改 + chain 完整」時 chain 能獨立偵測的情境。
- **GENESIS 作為首筆 prev 的驗證**（B1）→ 場景存在，但測試矩陣未包含「偽造 GENESIS：非首筆 entry 的 prev 設為 GENESIS」這個 N 類測試。

**建議：** 補充：
- N4：舊格式 log（無 h 欄）→ 不報 chain BLOCKER（確認容錯）
- N5：HWM 被竄改但 chain 完整 → chain 驗證通過（確認第二驗證的獨立性）
- N6：非首筆 entry 的 prev 偽設為 GENESIS → BLOCKER（確認 GENESIS 只允許首筆）

---

### FIND-5（嚴重度：LOW）：S3 Scenario Outline 的 Examples 混合了「技術狀態」與「業務狀態」

S3 Examples 中：
- `首筆 GENESIS` → 技術內部狀態（hash chain 結構）
- `舊格式未遷移` / `舊格式已遷移` → 遷移狀態

這些狀態對應 Given 的初始化方式完全不同，但 Outline 用同一個 `When chain 狀態為 "<state>"` 包含兩種本質不同的輸入類型，測試實作時會造成混淆。建議拆為兩個獨立 Scenario（GENESIS 邊界、舊格式兩個遷移狀態）。

建議：FIND-1 與 FIND-2 在 SPEC 實作前補齊；其餘 WARN 於實作中一同處理，不阻擋 SPEC 進入 G3。

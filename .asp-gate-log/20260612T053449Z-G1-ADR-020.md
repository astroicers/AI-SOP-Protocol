## 4. 對抗式檢驗核心主張

### F1（WARN）：能力邊界界定——誠實，但語意義務「留為殘留」缺後續追蹤機制

ADR 宣稱「語意型義務誠實留為殘留 + 後續追蹤」，但後續追蹤清單中對語意型義務僅一條：「P3：CONTEXT.md 術語檢查併入 G1/G2 gate subagent checklist」。Assumption Checkpoint、需求變更分級等其他語意義務**沒有任何追蹤機制**。「誠實承認殘留」是好的，但「殘留」若無追蹤入口，長期等同於放棄。建議至少建立一個 GitHub Issue 或 tech-debt 條目統一追蹤未機械化的 (c) 層語意義務。

### F2（WARN）：選項 B 否決理由「enforcement theater」成立，但論證未完整封閉

ADR 說 B 對語意義務無效。這個論點成立——「Assumption Checkpoint」確實無法用 hook 機械判定。但 ADR 沒有討論：如果只對**可機械判定的義務**套用 B 的「硬強制所有 hook」做法，B 和 C 實質上是否收斂？這個比較缺失，使得 B 的否決看起來略嫌倉促。實際上 C 已是 B 的子集（只機械化可判定義務），論述可更清楚。不構成阻擋，但論證需補強。

### F3（PASS）：「本 session 5 次漏跑 asp-ship」活證據的真實性

已驗證：`.asp-bypass-log.json` 顯示 11 筆記錄中，2026-06-12 當天有 1 筆（Step7），2026-06-11 有 8 筆（Step6/7 重複），`.asp-bypass-log.ndjson`（遷移後版本）只有 2 筆且均為 2026-04-09。git log 顯示 PR #29–#33 均為 2026-06-12 commits（b896f03 等），且 bypass log 中本 session 的 asp-ship 跳過痕跡不對應 5 個 PR 數量。ADR 說「本 session 那 5 次一筆未留」——這與 bypass log 實際狀態一致（bypass log 未記錄這些跳過）。**「乾淨的 bypass log 是倖存者偏差」這個診斷站得住，是本 ADR 最有力的論據。**

### F4（WARN）：PreToolUse hook 引入的新風險——ADR 有列但判定標準未具體化

ADR 列出「ship-trace 判定不可靠會誤擋」並要求提供 escape hatch。但「新鮮度判定」的具體標準缺失（多少分鐘算新鮮？staged 檔案變動後 ship-trace 是否自動失效？）。SPEC-013 存在且引用了 `.asp-test-result.json` + `.git/index` mtime，但 ADR 本身未交代這個關鍵判定邏輯的選擇依據。這會讓實作者有空間自行詮釋，增加誤擋風險。建議 ADR Decision 節加一條「新鮮度判定原則：以 `.git/index` mtime 為基準，ship-trace 必須在最後一次 git add 之後」之類的明確約束。

### F5（PASS）：與 ADR-019 一致性

已驗證 ADR-019 的兩個原則：
1. 「誠實界定能力邊界」——ADR-020 沿用，且 ADR-019 對「重算假 chain」的誠實殘留與 ADR-020 對「語意義務無法 hook」的誠實殘留結構相同。✓
2. 「hook 腳本納入 Iron Rule A」——ADR-020 Decision 第 2 條明確要求，Consequences 也重申，後續追蹤清單有 checkbox。✓

一致性：通過。

### F6（WARN）：死鎖場景的 escape hatch——有列但未測死鎖 edge case

ADR 提供 `ASP_SHIP_OK=1` + `make asp-unlock-commit` 兩個 escape hatch。但未討論：PreToolUse hook 自身若 exit 非零（hook 腳本有 bug / jq 未安裝 / 環境變數解析失敗）時的 fail-open vs fail-close 策略。ADR-019 bypass-hash 機制曾因「看守者」腳本自身問題引發過教訓（ADR-019 背景）。本 ADR 應明確宣告 hook 腳本 crash 時的 fallback 行為（建議 fail-open + WARN，而非 fail-close 完全擋住開發）。

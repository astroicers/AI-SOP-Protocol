# ASP v3.7 → v4.0 基線量測報告

> 量測時間：2026-04-29 | Branch: main | ASP 版本：3.7.0
> 機器可讀版本：`.asp-baseline-v3.7.json`

---

## 執行摘要

1. **CLAUDE.md 過胖**：309 行、16,598 字元，估算 ~4,500 tokens。其中 65 行是 Makefile 速查表（已可透過 `make help` 取得），30 行是 `validate_profile_config()` pseudocode（已在 `session-audit.sh` 實作），兩者合計 95 行是純 token 浪費。

2. **task_orchestrator.md 是最大瓶頸**：1,379 行，佔總 profile token 的約 21%。其中 handoff 模板邏輯、team-pick 推薦邏輯均為「被呼叫才執行」的 capability，適合轉為 skill——每次 session 不必全量載入。

3. **8 段 profile 邏輯具備 skill 化條件**：global_core.md 的 4 段（Fact Verification Gate、Assumption Checkpoint、Bug Classify、Change Cascade）、task_orchestrator.md 的 2 段（Handoff、Team Pick）、escalation.md 整體、dev_qa_loop.md 整體。這些均有明確觸發詞、一次性執行、無跨 session 狀態需求。

4. **L0 缺位**：ASP 從 L1 開始，沒有「探索性原型」等級。PoC 驗證、CYBERSEC 演講 demo、新技術可行性評估等場景，使用者被迫在 L1 的治理負擔下做探索性工作，摩擦感高。

5. **零 telemetry**：目前無法量測「哪些 rule 真的抓到問題」「哪些 gate 從未失敗」。v4.0 的 JSONL telemetry 系統是 evidence-based 持續改進的基礎。

6. **ASP 本身無威脅模型**：v3.7 的鐵則保護的是「使用 ASP 的專案」，但 ASP 框架本身（hooks、bypass log、session briefing）的攻擊面從未被正式審計。CYBERSEC 2026 演講需要這份材料。
